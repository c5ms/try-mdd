# 设计准则

> AI 推演时必须遵循的设计规约

---

## 项目定位

- **类型**：企业级
- **复杂度**：高

---

## 设计准则

### 一、数据一致性

- 聚合内强一致，聚合间最终一致
- 跨聚合操作通过领域事件解耦
- 关键业务操作需要事务保障

### 二、并发处理

- 关键操作需要并发控制（订单、库存、财务）
- 使用乐观锁机制处理并发冲突
- 高并发场景考虑分布式锁

### 三、权限控制

- 基于角色的权限控制（RBAC）
- 功能级权限控制
- 数据级权限隔离

### 四、业务合规

- 操作日志记录
- 审批流程
- 数据审计追踪

### 五、多币种

- 支持多币种交易与结算
- 币种配置管理
- 交易币种与结算币种分离

### 六、多汇率

- 实时/历史汇率管理
- 支持汇率波动核算
- 汇率来源可配置

### 七、正交性
- 状态设计要正交，不要将所有状态设计到一个属性中
- 所有业务规则要正交，不可以一个业务规则包含2个职责，规则可以编排

## DDD 建模规约

### 一、DDD 建模语言

#### 1. 核心概念

| 概念 | 定义 | 说明 |
|------|------|------|
| 领域 | 业务问题空间 | 业务的整体范围 |
| 子域 | 领域的细分 | 核心域、支撑域、通用域 |
| 限界上下文 | 领域在系统中的边界 | 模型的边界划分 |

#### 2. 建模要素

| 要素 | 特征 | 判断标准 |
|------|------|----------|
| 实体 | 有唯一标识 | 需要追踪生命周期 |
| 聚合 | 一组相关对象的边界 | 保持一致性边界 |
| 领域事件 | 领域中发生的事情 | 状态变更触发 |

**禁止设计值对象：**

| 禁止内容 | 禁止原因 |
|----------|----------|
| 值对象章节 | 过度DDD抽象，对大模型推演价值低 |
| 值对象定义 | 大模型推演不需要区分实体/值对象，仅需知道属性 |
| 值对象判断逻辑 | 增加设计复杂度，降低推演效率 |

**替代方案：**

值对象概念改为**结构化属性**，直接作为实体属性：

```markdown
| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| address | object | | 地址（结构化属性：street、city、state、postcode、country） |
```

---

### 二、领域边界划分

#### 1. 领域分类

| 类型 | 定义 | 优先级 |
|------|------|--------|
| 核心域 | 业务核心竞争力 | 最高 |
| 支撑域 | 辅助核心域的功能 | 中 |
| 通用域 | 可复用的通用功能 | 低 |

#### 2. 限界上下文划分原则

- **业务语言一致性**：上下文内术语统一
- **团队协作边界**：与团队职责对齐
- **模型独立性**：上下文间模型可独立演进

#### 3. 上下文映射模式

| 模式 | 定义 | 适用场景 |
|------|------|----------|
| 共享内核 | 上下文间共享部分模型 | 紧密协作的上下文 |
| 客户方/供应方 | 下游依赖上游模型 | 上游模型稳定 |
| 防腐层 | 下游隔离上游模型变化 | 上游模型不稳定 |

---

### 三、统一语言定义

#### 1. 术语表原则

- 业务术语与技术术语对齐
- 领域专家与设计师共用词汇
- 术语必须在上下文内一致

#### 2. 术语定义格式

```markdown
| 术语 | 英文名 | 定义 | 关联术语 |
|------|--------|------|----------|
| {中文} | {English} | {定义} | {关联术语} |
```

---

### 四、建模设计约束

#### 1. 聚合边界约束

| 约束 | 说明 |
|------|------|
| 聚合间引用 | 通过 ID 引用，不直接持有对象 |
| 聚合内部一致性 | 聚合边界内保持一致性 |
| 聚合根访问入口 | 聚合根是唯一的访问入口 |
| 聚合大小 | 适度，避免过大 |

#### 2. 领域事件命名

| 规则 | 示例 |
|------|------|
| 格式：{动词} + {名词} + {过去式} | OrderCreated、PaymentCompleted |
| 携带必要上下文，不含完整实体 | orderId、amount |

---

### 五、推演约束

#### 1. 规约优先级

| 优先级 | 规约类型 |
|--------|----------|
| 1（最高） | 用户红线（用户原话） |
| 2 | 领域规约（domain/{domain}/model.md） |
| 3 | 全局规约（config/rules.md） |

#### 2. 术语一致性

- 推演产出必须使用统一语言中的术语
- 新增术语必须补充到领域术语表

#### 3. 边界完整性

- 聚合边界必须在领域规约中明确定义
- 跨聚合操作必须通过领域事件解耦

#### 4. 禁止设计值对象

**强制约束：**

| 约束 | 说明 | 验收标准 |
|------|------|----------|
| **禁止值对象章节** | 领域模型文件不允许包含"值对象"章节 | ✅ 无值对象章节 ❌ 有值对象章节 |
| **禁止值对象定义** | 不允许设计独立的值对象实体 | ✅ 值对象合并为实体属性 ❌ 独立值对象实体 |
| **禁止值对象判断** | 不允许输出实体vs值对象判断逻辑 | ✅ 无判断章节 ❌ 有判断章节 |

**替代方案：**

值对象概念改为**结构化属性**，格式如下：

```markdown
| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| address | object | | 地址（结构化属性：street、city、state、postcode、country） |
| inspectionRequirement | object | | 验货要求（结构化属性：requirementType、description） |
```

**禁止原因：**

| 原因 | 说明 |
|------|------|
| 过度DDD抽象 | 值对象是DDD概念，对大模型推演价值低 |
| 降低推演效率 | 大模型推演不需要区分实体/值对象 |
| 增加设计复杂度 | 值对象判断逻辑增加设计成本 |
| 污染模型文档 | 值对象章节会污染"大模型内存" |

---

### 六、规约约束机制

| 规则 | 触发时机 | 验收标准 | 说明 |
|------|----------|----------|------|
| 推演前读取 | `/m-design` 启动时 | 已读取 rules.md + roles.md | 规约是设计依据 |
| 角色必须参与 | 方案输出前 | 有角色评审汇总章节 | 单视角设计会遗漏场景 |
| 用户原话保留 | 方案输出时 | 有用户原话章节 | 用户原话是设计依据 |
| 用户确认 | 更新设计前 | 有 AskUserQuestion 步骤 | 绕过用户决策权 |
| 聚合间引用 | 聚合设计时 | 通过 ID 引用 | 违反聚合边界原则 |
