# MDD：在写代码前，先完成领域模型推演

## 这是什么？

一句话：**需求 → 角色评审推演 → 方案确认 → 开发**

不是写代码的工具，是写代码前验证设计的工具。

传统的流程是：需求 → 直接开发 → 发现问题 → 返工

MDD 的流程是：需求 → 设计推演 → 角色评审 → 方案确认 → 开发

核心区别：**在写代码前，把设计问题暴露出来并解决掉。**

## 解决什么问题？

开发过复杂业务系统的，都遇到过这些问题：

| 痛点 | 后果 |
|------|------|
| 业务人员口头描述需求，开发人员理解偏差 | 做出来的东西不是业务要的，返工 |
| 设计遗漏场景，到代码阶段才发现 | 重构成本高，改一个地方影响一片 |
| 业务和开发沟通语言不统一 | 同一个词，两边理解不一样，歧义频出 |
| 不确定事项被遗忘 | 需求丢失，上线后才发现问题 |
| 单视角设计，考虑不全面 | 只看了技术视角，忽略了财务、物流、生产等视角 |

这些问题有个共同点：**发现得太晚。**

代码写完了才发现需求理解错，重构成本是设计阶段修改的10倍以上。

MDD 要做的，就是**把问题暴露在设计阶段，而不是代码阶段。**

## 怎么解决？

MDD 提供了四个核心机制：

### 1. 角色评审机制

不是 AI 单独输出方案，而是 AI 扮演多个角色，从各自视角评审方案。

比如设计一个物料管理模块：

- **业务经理** 关注：物料信息完整性、流程合理性
- **财务主管** 关注：计量单位换算、成本关联
- **生产跟单员** 关注：物料分类是否便于生产管理
- **系统架构师** 关注：聚合边界是否清晰、扩展性如何

每个角色从自己的视角挑问题、提建议，补充遗漏场景。

**解决什么问题？** 单视角设计的盲区，多人评审才能暴露。

### 2. 统一术语表

业务人员和开发人员使用同一份术语表，术语定义清晰明确。

```markdown
| 术语 | 英文名 | 定义 | 关联术语 |
|------|--------|------|----------|
| 物料 | Material | 生产所需的原料、辅料等基础物资 | 物料分类 |
| 物料分类 | MaterialCategory | 物料的类别划分 | 物料 |
```

**解决什么问题？** 沟通歧义。"订单"在业务眼里和开发眼里可能是两个东西，术语表强制对齐。

### 3. 用户原话保留

用户原始需求原样保留在方案中：

```markdown
## 用户原话

> 物料字典需要包含：基本信息（编号、名称、规格）、物料分类、计量单位、供应商关联
```

**解决什么问题？** 设计追溯。当方案和需求有出入时，可以追溯到用户原话确认。

### 4. 待定问题独立追踪

不确定的事项，记录到独立的待定清单：

```markdown
| # | 待定内容 | 来源 | 状态 |
|---|----------|------|------|
| 1 | 物料编码具体生成规则 | 架构师评审 | 待确认 |
| 2 | 物料图片字段设计 | 生产跟单员评审 | 待确认 |
```

**解决什么问题？** 不遗忘。不确定事项不会在设计过程中被忽略，而是显式追踪。

## 什么时候用？

不是所有项目都需要 MDD。

| 场景 | 用不用 |
|------|--------|
| 简单 CRUD，需求清晰 | 不用 |
| 个人项目，自己就是业务方 | 不用 |
| 业务复杂，多方角色关注 | **用** |
| 需求模糊，需要反复确认 | **用** |
| 团队沟通成本高，术语混乱 | **用** |
| 企业级系统，重构成本高 | **用** |

判断标准：**需求理解偏差的代价有多大？**

如果需求理解偏差只是改几行代码的事，不需要 MDD。

如果需求理解偏差意味着重构核心模块、影响多个下游系统，那就需要 MDD。

## 用了有什么好处？

直接对比：

| 维度 | 不用 MDD | 用 MDD |
|------|----------|--------|
| 返工成本 | 代码阶段发现问题 → 重构 | 设计阶段发现问题 → 改方案 |
| 沟通效率 | 反复确认、歧义频出 | 统一术语、共享模型 |
| 遗漏场景 | 代码阶段暴露 | 角色评审提前暴露 |
| 设计追溯 | 需求原文丢失，无法追溯 | 用户原话保留，有据可查 |
| 不确定项 | 被遗忘 | 显式追踪，不遗漏 |

简单说：**用 MDD 的成本是设计阶段多花点时间，收益是降低返工成本。**

## 怎么用？

MDD 是一个命令行工具，基于 AI 对话。

### 1. 初始化项目

```bash
/m-init
```

AI 会问你几个问题，然后生成：
- `config/rules.md` — 设计准则（DDD 规约、项目定位）
- `config/roles.md` — 角色定义（业务经理、财务主管、架构师等）
- 目录结构

### 2. 设计领域模型

```bash
/m-design 订单管理
```

然后描述你的需求：

```
用户: /m-design 订单管理
用户: 订单属于客户，一个客户可以有多个订单。订单有状态：待确认、生产中、已完成...
```

AI 执行的流程：

```
读取规约 → config/rules.md（DDD 建模规约）
    ↓
确定角色 → config/roles.md（哪些角色参与评审）
    ↓
明确需求 → 理解用户输入，整理成结构化需求
    ↓
角色评审 → 业务经理、财务主管、架构师逐角色评审
    ↓
输出方案 → solution/order.md（待确认方案）
    ↓
用户确认 → 是否更新到领域模型？
```

输出的方案包含：
- 术语表
- 实体定义
- 聚合边界
- 状态机
- 业务规则
- 角色评审汇总
- 待定项清单

### 3. 应用方案

确认方案没问题后：

```bash
/m-apply
```

方案会写入 `design/domain/order.md`，成为正式的领域模型文件。

同时更新 `design/model.md`（领域模型汇总）和 `design/todo.md`（待定问题追踪）。

### 4. 讨论/推理

设计过程中有疑问：

```bash
/m-evolve 订单状态应该如何设计？生产中和已完成之间是否需要质检状态？
```

AI 会帮你推理，给出分析，理清思路。

### 5. 反思改进

发现设计有问题：

```bash
/m-reflect
```

AI 会帮你梳理错误、分析根因、给出改进建议。

## 目录结构

```
config/
  rules.md          # 设计准则（DDD 建模规约、项目定位）
  roles.md          # 角色定义（参与设计的角色）

solution/
  {domain}.md       # 待确认方案（角色评审后）

design/
  todo.md           # 待定问题追踪
  model.md          # 领域模型汇总（统一术语表、领域关系图）
  domain/
    {domain}.md     # 领域模型（实体、聚合、状态机、规则）
```

**流程：**

1. `solution/` 放待确认方案
2. 用户确认后，`/m-apply` 写入 `design/domain/`
3. `design/model.md` 汇总所有领域，形成统一术语表

## 核心设计原则

MDD 遵循 DDD（领域驱动设计）规约：

| 规则 | 说明 |
|------|------|
| 聚合间引用 | 通过 ID 引用，不直接持有对象 |
| 用户原话保留 | 用户原始需求原样保留在方案中 |
| 角色必须参与 | 每个方案必须经过角色评审 |
| 待定项追踪 | 不确定事项记录到独立清单 |
| 用户确认 | 更新设计前必须用户确认 |

这些规则硬编码在框架中，AI 必须遵守。

## 与其他工具的区别

| 工具 | 定位 | 解决什么问题 |
|------|------|--------------|
| MDD | 设计阶段 | 需求理解偏差、设计遗漏 |
| SDD（Spec-Driven） | 开发阶段 | 代码落地效率 |
| 原型工具 | 验证阶段 | 用户体验验证 |

MDD 在原型之前，SDD 在设计之后。

**MDD → 原型 → SDD → 代码**

## 总结

MDD 的价值主张：**在设计阶段暴露问题，而不是代码阶段。**

核心机制：
- 角色评审 → 多视角补充遗漏
- 统一术语表 → 消除沟通歧义
- 用户原话保留 → 设计可追溯
- 待定项追踪 → 不遗忘

适用场景：业务复杂、需求模糊、多方角色关注、重构成本高的系统。

不适用场景：简单 CRUD、需求清晰、个人项目。

---

**一句话总结：如果你的项目需求理解偏差代价很高，MDD 可以帮你降低这个风险。**
