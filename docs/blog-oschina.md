# MDD：让大模型像下象棋一样推演你的业务决策

## 这是什么？

一句话：**没有代码，也能推演程序运行结果。**

就像 AI 下象棋：模拟下一步会对未来多步有什么影响。

MDD 让你：只说"如果xxx会怎样xxx"，大模型模拟运行，判断你的想法是否可行。

**核心定位：降低试错成本。**

以前有想法不敢试，因为人工验证成本太高。现在让大模型推演，成本几乎为零。

## 解决什么问题？

| 痛点 | 后果 |
|------|------|
| 有想法不敢试 | 试错成本太高，人工验证代价大，好想法被放弃 |
| 需求变更不敢拍板 | 不知道改了会影响什么，怕改出问题 |
| 业务规则靠人记 | 忘了就问人，沟通效率低，容易出错 |
| 大模型逐块开发 | 短路，前后不连贯，做了一块忘了上一块 |
| 业务和开发沟通成本高 | 同一个词两边理解不一样，反复确认 |

这些问题有个共同点：**验证成本高。**

要么不敢试，要么试了代价大。

MDD 要做的，就是**把验证成本降到几乎为零。**

## 怎么解决？

### 1. 无代码推演

只说"如果xxx会怎样"，大模型模拟程序运行。

例子：

```
用户: 如果我把订单的"待确认"状态改成直接跳到"生产中"，会怎样？

AI: 模拟推演：
  步骤1: 订单创建后直接进入生产中状态
  步骤2: 生产计划自动触发，生成物料需求
  步骤3: 物料需求触发采购申请
  步骤4: 但问题来了——客户还没确认订单，可能取消
  步骤5: 如果取消，生产已经启动，物料已采购
  步骤6: 导致库存积压、成本浪费
  
  结论：不可行。建议保留"待确认"状态，客户确认后再触发生产。
```

**没有一行代码，你就知道这个想法行不行。**

就像 AI 下象棋，模拟下一步，看到五步之后的后果。

### 2. 大模型作为调研-设计-论证助手

大模型知识比人类多，这套方法把它流程化：

| 阶段 | 大模型做什么 |
|------|--------------|
| 调研 | 搜索行业实践、对比方案、补充背景知识 |
| 设计 | 基于规约输出领域模型、实体、状态机 |
| 论证 | 模拟运行、暴露问题、提出建议 |

**不是替代你，而是辅助你。**

你做决策，大模型帮你验证。

### 3. 可进化系统

你纠正大模型的错误 → 反思技能自动更新到 skill/doc/config。

系统越来越懂你，越来越懂你的业务。

例子：

```
用户: 你刚才说物料需求=订单数量×单件用量，这个不对，还要考虑损耗率。

AI: 收到纠正，触发反思技能：
  更新 config/rules.md → 物料需求计算规则
  更新 design/domain/production-rule.md → 损耗率参数
  下次推演自动应用新规则
```

**你的纠正不会丢失，系统会记住。**

### 4. 业务规则随时查

忘记了某个业务规则？直接问：

```
用户: 生产物料需求计算公式是什么？

AI: 根据设计规约：
  物料需求 = 订单数量 × 单件用量 × (1 + 损耗率) + 安全余量
  计算时机：订单确认后触发
  触发事件：OrderConfirmed
```

秒回，快速跟财务部门对齐设计。

**不需要翻文档，不需要问人，直接问大模型。**

### 5. 设计产物可加工

设计完成不是终点，可以输出：

| 产物 | 说明 |
|------|------|
| 流程图 | 所有用例的流程图 |
| 原型 | 快速实现系统原型用于展示 |
| 开发计划 | 拆分领域、拆分开发步骤 |
| 测试用例 | 基于业务规则生成测试场景 |

### 6. 全局规划，避免短路

设计完成 → 论证流程通顺 → 大模型拆分开发步骤。

避免大模型逐块开发时前后不连贯。

```
AI 输出开发计划：
  第1步: 完成基础数据领域（物料、单位、分类）
  第2步: 完成客户管理领域
  第3步: 完成订单管理领域（依赖基础数据）
  第4步: 完成生产管理领域（依赖订单）
  第5步: 完成采购管理领域（依赖生产）
  ...
```

**先规划，再执行，不会短路。**

## 什么时候用？

| 场景 | 用不用 |
|------|--------|
| 业务复杂，决策影响链条长 | **用** |
| 有想法不敢试，怕成本高 | **用** |
| 需求变更频繁，需要快速验证 | **用** |
| 大模型辅助开发，但前后不连贯 | **用** |
| 简单系统，需求清晰 | 不用 |
| 个人项目，自己就是业务方 | 不用 |

判断标准：**试错成本有多高？**

如果试错成本低（改几行代码的事），不需要 MDD。

如果试错成本高（影响多个下游系统、重构核心模块），那就需要 MDD。

## 用了有什么好处？

| 维度 | 不用 MDD | 用 MDD |
|------|----------|--------|
| 试错成本 | 高（写代码验证） | 几乎为零（推演） |
| 决策信心 | 不敢拍板，怕出问题 | 推演看后果，心中有数 |
| 业务规则查询 | 问人，效率低 | 问大模型，秒回 |
| 开发连贯性 | 逐块开发，短路 | 全局规划，前后连贯 |
| 系统进化 | 错误重复犯 | 反思自动更新，越来越懂你 |

简单说：**用 MDD 的成本是设计阶段多花点时间，收益是把试错成本降到几乎为零。**

## 怎么用？

MDD 是一个命令行工具，基于 AI 对话。

### 1. 初始化项目

```bash
/m-init
```

AI 会问你几个问题，然后生成：
- `config/rules.md` — 设计准则（DDD 规约、项目定位、业务规则）
- `config/roles.md` — 角色定义（业务经理、财务主管、架构师等）
- 目录结构

### 2. 设计领域模型

```bash
/m-design 订单管理
```

然后描述需求：

```
用户: /m-design 订单管理
用户: 订单属于客户，一个客户可以有多个订单。订单有状态：待确认、生产中、已完成...
```

AI 执行流程：

```
读取规约 → 理解业务规则
    ↓
确定角色 → 哪些角色参与评审
    ↓
明确需求 → 整理成结构化需求
    ↓
角色评审 → 多视角补充遗漏
    ↓
输出方案 → solution/order.md
    ↓
用户确认 → 是否更新？
```

### 3. 推演验证

有想法想验证：

```bash
/m-evolve 如果订单取消后，已经采购的物料怎么处理？
```

AI 模拟推演，给出分析。

### 4. 应用方案

确认方案没问题：

```bash
/m-apply
```

写入 `design/domain/order.md`，成为正式领域模型。

### 5. 反思改进

纠正大模型的错误：

```bash
/m-reflect
```

系统自动更新 skill/doc/config，下次推演应用新规则。

## 目录结构

```
config/
  rules.md          # 业务规则、设计准则
  roles.md          # 角色定义

solution/
  {domain}.md       # 待确认方案

design/
  todo.md           # 待定问题追踪
  model.md          # 领域模型汇总（统一术语表）
  domain/
    {domain}.md     # 领域模型（实体、聚合、状态机）
    {domain}-rule.md # 业务规则补充
```

## 核心机制

| 机制 | 做什么 | 解决什么 |
|------|--------|----------|
| 无代码推演 | 说"如果xxx会怎样"，大模型模拟运行 | 试错成本高，不敢试 |
| 角色评审 | 多视角补充遗漏场景 | 单视角设计的盲区 |
| 统一术语表 | 业务和开发共享词汇 | 沟通歧义 |
| 反思进化 | 纠错自动更新系统 | 错误重复犯 |
| 业务规则查询 | 直接问大模型 | 忘记规则，沟通效率低 |
| 全局规划 | 拆分开发步骤 | 逐块开发短路 |

## 与其他工具的区别

| 工具 | 定位 | 解决什么 |
|------|------|----------|
| MDD | 设计推演阶段 | 试错成本、验证可行性 |
| SDD | 开发阶段 | 代码落地效率 |
| 原型工具 | 展示阶段 | 用户体验验证 |

MDD 在原型之前，SDD 在设计之后。

**MDD → 原型 → SDD → 代码**

## 总结

MDD 的价值主张：**让大模型像下象棋一样推演你的业务决策，把试错成本降到几乎为零。**

核心机制：
- 无代码推演 → 不敢试的现在敢试了
- 业务规则查询 → 秒回，快速对齐
- 反思进化 → 系统越来越懂你
- 全局规划 → 避免开发短路

适用场景：业务复杂、决策影响链条长、试错成本高。

不适用场景：简单系统、需求清晰、试错成本低。

---

**一句话总结：如果你的项目试错成本高，不敢轻易尝试新想法，MDD 可以让你免费试错。**
