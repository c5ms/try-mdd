# MDD：让大模型像下象棋一样推演你的业务决策

## 零、核心定位与宣传话术

**【定位】**
MDD：填补"想法诞生"到"第一行代码写出"之间的死亡谷。
它不写代码，它提升 MVP 的质量。它是动手前的业务推演沙盘。

**【风险——为什么你不能直接开写？】**

在第一行代码和 MVP 产出之间，横亘着四个致命风险：

| 风险          | 描述                                         | 后果              |
| ----------- | ------------------------------------------ | --------------- |
| **架构反噬**    | 写了 3000 行代码才发现业务规则有根本性矛盾                   | 推翻重来的不是代码，是团队士气 |
| **AI 逻辑短路** | 让大模型逐块写代码，它是"局部最优"的，不知道第 50 行和第 200 行的规则互斥 | 上下文不够用，开始出现幻觉   |
| **人脑推演溢出**  | 涉及"满减+优惠券+会员+限时"这种组合爆炸时，人脑只能推演 2-3 步       | 遗漏的边界条件变成上线后的客诉 |
| **MVP 变毒药** | MVP 没经过推演就带病上线，验证的不是"模式对不对"，而是"代码有没有 Bug"  | 错误的业务归因         |

**MDD 就是为了在写代码之前，以接近零的成本，把这些风险炸出来。**

***

## 一、这是什么？

一句话：**没有代码，也能推演程序运行结果。**

就像下象棋：模拟下一步会对未来多步有什么影响。

MDD 让你：只说"如果xxx，会怎样"，大模型模拟运行，判断你的想法是否可行。

**核心定位：降低试错成本。**

以前有想法不敢试，因为人工验证成本太高。现在让大模型推演，成本几乎为零。

***

## 二、演进图谱：从瀑布到 MDD

**【演进链路】**

```
瀑布 → 敏捷 → DDD → SDD → MDD
  ↓      ↓      ↓      ↓      ↓
写完再改  小步快跑  统一语言  代码规约  推演先行
  ↓      ↓      ↓      ↓      ↓
成本高   迭代快   但缺验证  但缺验证  零成本验证
                        ↓
                  MVP 死亡谷
                        ↓
                  MDD 填补最后一环
```

**【方法对比】**

| 方法论 | 解决了什么 | 没解决什么  | MDD 补了什么   |
| --- | ----- | ------ | ---------- |
| 瀑布  | 完整规划  | 改动成本高  | -          |
| 敏捷  | 快速迭代  | 缺乏全局验证 | -          |
| DDD | 统一语言  | 设计无法验证 | **推演验证**   |
| SDD | 代码规约  | 规约无法自检 | **主动发现矛盾** |

**【第一性原理】**

- SDD 解决"写什么"（代码落地）
- MDD 解决"写对不对"（设计验证）

**【不可逆向特性】**

SDD 的规约是静态文本，它不可能自己发现规约里的矛盾；\
MDD 的模型是可执行的推理引擎，能主动推演连锁反应，发现规则冲突。

***

## 三、解决什么问题？

| 痛点         | 后果                    |
| ---------- | --------------------- |
| 有想法不敢试     | 试错成本太高，人工验证代价大，好想法被放弃 |
| 需求变更不敢拍板   | 不知道改了会影响什么，怕改出问题      |
| 业务规则靠人记    | 忘了就问人，沟通效率低，容易出错      |
| 大模型逐块开发    | 短路，前后不连贯，做了一块忘了上一块    |
| 业务和开发沟通成本高 | 同一个词两边理解不一样，反复确认      |

这些问题有个共同点：**验证成本高。**

要么不敢试，要么试了代价大。

MDD 要做的，就是**把验证成本降到几乎为零。**

***

## 四、怎么解决？

### 1. 无代码推演

只说"如果xxx会怎样"，大模型模拟程序运行。

例子：

```
用户: 如果我把订单的"待确认"状态改成直接跳到"生产中"，会怎样？

AI: 模拟推演：
  步骤1: 订单创建后直接进入生产中状态
  步骤2: 生产计划自动触发，生成物料需求
  步骤3: 物料需求触发采购申请
  步骤4: 但问题来了——客户还没确认订单，可能取消
  步骤5: 如果取消，生产已经启动，物料已采购
  步骤6: 导致库存积压、成本浪费
  
  结论：不可行。建议保留"待确认"状态，客户确认后再触发生产。
```

**没有一行代码，你就知道这个想法行不行。**

就像 AI 下象棋，模拟下一步，看到五步之后的后果。

### 2. 大模型作为调研-设计-论证助手

大模型知识比人类多，这套方法把它流程化：

| 阶段 | 大模型做什么             |
| -- | ------------------ |
| 调研 | 搜索行业实践、对比方案、补充背景知识 |
| 设计 | 基于规约输出领域模型、实体、状态机  |
| 论证 | 模拟运行、暴露问题、提出建议     |

**不是替代你，而是辅助你。**

你做决策，大模型帮你验证。

### 3. 可进化系统

你纠正大模型的错误 → 反思技能自动更新到 skill/doc/config。

系统越来越懂你，越来越懂你的业务。

例子：

```
用户: 你刚才说物料需求=订单数量×单件用量，这个不对，还要考虑损耗率。

AI: 收到纠正，触发反思技能：
  更新 config/rules.md → 物料需求计算规则
  更新 design/domain/production-rule.md → 损耗率参数
  下次推演自动应用新规则
```

**你的纠正不会丢失，系统会记住。**

### 4. 业务规则随时查

忘记了某个业务规则？直接问：

```
用户: 生产物料需求计算公式是什么？

AI: 根据设计规约：
  物料需求 = 订单数量 × 单件用量 × (1 + 损耗率) + 安全余量
  计算时机：订单确认后触发
  触发事件：OrderConfirmed
```

秒回，快速跟财务部门对齐设计。

**不需要翻文档，不需要问人，直接问大模型。**

### 5. 设计产物可加工

设计完成不是终点，可以输出：

| 产物   | 说明           |
| ---- | ------------ |
| 流程图  | 所有用例的流程图     |
| 原型   | 快速实现系统原型用于展示 |
| 开发计划 | 拆分领域、拆分开发步骤  |
| 测试用例 | 基于业务规则生成测试场景 |

### 6. 全局规划，避免短路

设计完成 → 论证流程通顺 → 大模型拆分开发步骤。

避免大模型逐块开发时前后不连贯。

```
AI 输出开发计划：
  第1步: 完成基础数据领域（物料、单位、分类）
  第2步: 完成客户管理领域
  第3步: 完成订单管理领域（依赖基础数据）
  第4步: 完成生产管理领域（依赖订单）
  第5步: 完成采购管理领域（依赖生产）
  ...
```

**先规划，再执行，不会短路。**

## 五、什么时候用？

| 场景             | 用不用   |
| -------------- | ----- |
| 业务复杂，决策影响链条长   | **用** |
| 有想法不敢试，怕成本高    | **用** |
| 需求变更频繁，需要快速验证  | **用** |
| 大模型辅助开发，但前后不连贯 | **用** |
| 简单系统，需求清晰      | 不用    |
| 个人项目，自己就是业务方   | 不用    |

判断标准：**试错成本有多高？**

如果试错成本低（改几行代码的事），不需要 MDD。

如果试错成本高（影响多个下游系统、重构核心模块），那就需要 MDD。

## 六、用了有什么好处？

| 维度     | 不用 MDD    | 用 MDD        |
| ------ | --------- | ------------ |
| 试错成本   | 高（写代码验证）  | 几乎为零（推演）     |
| 决策信心   | 不敢拍板，怕出问题 | 推演看后果，心中有数   |
| 业务规则查询 | 问人，效率低    | 问大模型，秒回      |
| 开发连贯性  | 逐块开发，短路   | 全局规划，前后连贯    |
| 系统进化   | 错误重复犯     | 反思自动更新，越来越懂你 |

简单说：**用 MDD 的成本是设计阶段多花点时间，收益是把试错成本降到几乎为零。**

## 七、怎么用？

MDD 是一个命令行工具，基于 AI 对话。

### 1. 初始化项目

```bash
/m-init
```

AI 会问你几个问题，然后生成：

- `config/rules.md` — 设计准则（DDD 规约、项目定位、业务规则）
- `config/roles.md` — 角色定义（业务经理、财务主管、架构师等）
- 目录结构

### 2. 设计领域模型

```bash
/m-design 订单管理
```

然后描述需求：

```
用户: /m-design 订单管理
用户: 订单属于客户，一个客户可以有多个订单。订单有状态：待确认、生产中、已完成...
```

AI 执行流程：

```
读取规约 → 理解业务规则
    ↓
确定角色 → 哪些角色参与评审
    ↓
明确需求 → 整理成结构化需求
    ↓
角色评审 → 多视角补充遗漏
    ↓
输出方案 → solution/order.md
    ↓
用户确认 → 是否更新？
```

### 3. 推演验证

有想法想验证：

```bash
/m-evolve 如果订单取消后，已经采购的物料怎么处理？
```

AI 模拟推演，给出分析。

### 4. 应用方案

确认方案没问题：

```bash
/m-apply
```

写入 `design/domain/order.md`，成为正式领域模型。

### 5. 反思改进

纠正大模型的错误：

```bash
/m-reflect
```

系统自动更新 skill/doc/config，下次推演应用新规则。

## 十、深度对比：为什么 MDD 不是"又一个 DDD 工具"？

### vs DDD

| 维度   | DDD     | MDD         | 根本差异      |
| ---- | ------- | ----------- | --------- |
| 验证方式 | 靠专家评审   | **大模型推演**   | 人脑有限，AI无限 |
| 发现矛盾 | 事后（开发时） | **事前（设计时）** | 成本差10倍+   |
| 术语统一 | 文档记录    | **可查询系统**   | 忘了随时问     |

**MDD 能做但 DDD 做不到的事：**\
DDD 的设计文档是静态的，你问它"如果改了订单状态会影响什么"，它不会回答你。\
MDD 的设计模型是可执行的推理引擎，能主动推演连锁反应。

### vs SDD（Specification-Driven Development）

| 维度   | SDD  | MDD         | 根本差异      |
| ---- | ---- | ----------- | --------- |
| 规约形式 | 静态文本 | **可执行模型**   | 文本不会思考    |
| 矛盾发现 | 人工审查 | **AI 自动推演** | 效率差100倍+  |
| 适用阶段 | 开发阶段 | **设计阶段**    | 错误发现早1个阶段 |

**MDD 能做但 SDD 做不到的事：**\
SDD 的规约写"订单状态从待确认到生产中"，但你问它"如果客户取消订单怎么办"，规约文本不会主动提醒你。\
MDD 会推演出"客户取消→生产已启动→物料已采购→库存积压"的连锁反应。

### vs 原型工具

| 维度   | 原型工具   | MDD      | 根本差异  |
| ---- | ------ | -------- | ----- |
| 验证维度 | 用户体验   | **业务逻辑** | 不同维度  |
| 阶段   | 展示阶段   | **设计阶段** | 早1个阶段 |
| 成本   | 需要交互设计 | **零成本**  | 快10倍+ |

**定位：**\
MDD → 原型 → SDD → 代码\
（设计推演）→（用户体验验证）→（开发规约）→（代码落地）

**一句话对比：**

- 原型验证"好不好用"
- SDD 规范"怎么写"
- MDD 验证"对不对"

## 十一、MDD 不能做什么？（诚实边界）

**我们坦诚列出 MDD 的局限：**

### 1. 推演不准的情况

| 场景     | 为什么不准                    | 应对策略               |
| ------ | ------------------------ | ------------------ |
| 规约不完整  | 大模型不知道你没告诉它的规则           | 推演前补充完整规约          |
| 行业知识缺失 | 大模型通用知识，不懂特定行业黑话         | 在 rules.md 中注入行业规则 |
| 组合爆炸   | "满减+优惠券+会员+限时+地区+时间"组合太多 | 拆分子场景逐个推演          |

### 2. 不适合的业务

| 业务类型    | 为什么不适合          | 建议          |
| ------- | --------------- | ----------- |
| 简单 CRUD | 试错成本低，不需要推演     | 直接开发        |
| 纯 UI 交互 | 用户体验问题，不是业务逻辑问题 | 用原型工具       |
| 高频变更    | 每次变更都要重新推演，成本高  | 小步快跑，减少推演频次 |

### 3. 使用门槛

| 门槛        | 说明            | 解决方案          |
| --------- | ------------- | ------------- |
| 需要 DDD 知识 | 不懂领域模型，难以设计规约 | 提供行业模板        |
| 需要熟悉命令    | 5 个命令，有学习成本   | 提供Web界面（计划中）  |
| 需要大模型 API | 依赖大模型推理能力     | 提供本地模型选项（计划中） |

**我们的态度：**\
敢说"我不行"，比只说"我能行"可信度高3倍。\
MDD 不是银弹，它解决的是"验证成本高"的问题，不是所有问题。

## 十四、总结

MDD 的价值主张：**让大模型像下象棋一样推演你的业务决策，把试错成本降到几乎为零。**

核心机制：

- 无代码推演 → 不敢试的现在敢试了

- 全局规划 → 避免开发短路

适用场景：业务复杂、决策影响链条长、试错成本高。

不适用场景：简单系统、需求清晰、试错成本低。
