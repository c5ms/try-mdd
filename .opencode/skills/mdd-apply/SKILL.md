# MDD 方案应用

本 skill **不自动触发**，需用户通过 `/m-apply` command 调用。

---

## 核心理念

```
搜索方案 → 用户确认 → 执行更新 → 更新汇总 → 更新待定问题 → 清理方案
```

---

## 输入

由 `/m-apply` command 触发，无需额外参数。

---

## 步骤

### Step 1：搜索所有方案

搜索 `solution/` 目录下所有的方案文件：

```
solution/{domain}.md
```

**搜索方式**：使用 Glob 工具搜索 `solution/*.md`

**领域名称提取**：方案文件名去掉 `.md` 后缀即为领域名

**输出格式**：

```
发现待应用方案：

| # | 领域 | 方案文件 |
|---|------|----------|
| 1 | {domain1} | solution/{domain1}.md |
| 2 | {domain2} | solution/{domain2}.md |
```

---

### Step 2：用户确认

使用 **AskUserQuestion** 询问用户：

> 请选择要更新到 design/domain 目录的方案：

**选项设置**：
- `multiple: true` —— 允许多选
- 每个选项为方案文件名去掉 `.md` 后缀（如：customer、order）

**用户选择后**：
- 如果选择「全部应用」—— 应用所有方案
- 如果选择部分方案 —— 仅应用选中的方案
- 如果选择「取消」—— 不执行任何操作

---

### Step 3：执行更新

对用户确认的每个方案：

#### 3.1 读取方案文件

读取 `solution/{domain}.md`

#### 3.2 解析方案内容

从方案中提取：
- 术语表
- 实体定义
- 值对象
- 聚合边界
- 状态机（如有）
- 业务规则
- 数据流图（如有）
- 业务场景（如有）
- 待定项

#### 3.3 大模型判断拆分

**判断维度**：
- 内容复杂度（状态流转、规则逻辑、场景分支）
- 业务重要性（核心流程是否需要单独阐述）
- 可维护性（是否便于后续迭代）
- 关联关系（是否涉及多领域交互）

**拆分决策**：

| 补充类型 | 复杂时拆分 | 简单时处理 |
|----------|------------|------------|
| 状态机 | `{domain}-state.md` | 合并在主文件 |
| 业务规则 | `{domain}-rule.md` | 合并在主文件 |
| 数据流图 | `{domain}-flow.md` | 合并在主文件 |
| 业务场景 | `{domain}-scene.md` | 合并在主文件 |

**注意**：不做量化指标定义，由大模型根据专业知识判断。

#### 3.4 生成领域模型文件

**主文件**：`design/domain/{domain}.md`

**补充文件**（如拆分）：
- `design/domain/{domain}-state.md`
- `design/domain/{domain}-rule.md`
- `design/domain/{domain}-flow.md`
- `design/domain/{domain}-scene.md`

#### 3.5 写入领域模型

将生成的模型写入对应文件。

---

### Step 4：更新汇总文件

更新成功后，更新 `design/model.md`：

**汇总内容**：
- 重新扫描所有 `design/domain/*.md`
- 更新统一术语表（合并所有领域术语）
- 更新领域模型图（Mermaid）
- 更新聚合间引用表
- 更新领域概览（状态、文件列表）
- 更新演进记录（追加本次变更）

---

### Step 5：更新待定问题

更新 `design/todo.md`：

**处理逻辑**：
- 从方案中提取待定项 → 追加到"一、待确认问题"
- 如果方案解决了已有问题 → 从"待确认"移到"已解决问题"

---

### Step 6：清理方案文件

更新成功后，删除已应用的方案文件：

- 删除 `solution/{domain}.md`

---

## 输出

完成后输出：

```
应用完成：

已更新：
- design/domain/{domain1}.md         —— {领域1}领域 - {模型}模型
- design/domain/{domain1}-state.md   —— 状态机补充（如有）
- design/domain/{domain1}-rule.md    —— 业务规则补充（如有）
- design/model.md                    —— 领域模型汇总
- design/todo.md                     —— 待定问题追踪

已清理：
- solution/{domain1}.md
```

---

## 约束

- **用户确认**：必须经用户确认后才执行更新
- **顺序执行**：按用户选择的顺序逐个应用
- **失败处理**：如果某个方案更新失败，跳过并记录错误，继续处理其他方案
- **清理时机**：只有更新成功后才删除方案文件
- **汇总更新**：每次应用方案后必须更新 `design/model.md`
- **待定问题同步**：每次应用方案后必须同步 `design/todo.md`
- **平铺结构**：领域模型文件平铺在 design/domain/ 目录
- **拆分判断**：大模型根据复杂度判断是否拆分补充文件，不做量化指标定义
