# 建模元规则

> 大模型设计输出前的强制约束——来自40年软件工程沉淀

---

## 一、法则层：五条核心法则

### 法则1：状态正交分解

**核心法则**：一个状态字段只能描述一个独立问题空间。

**理论背景**：Harel Statecharts（1987）——AND状态（正交并行）vs OR状态（互斥）。

| 禁止事项 | 禁止原因 |
|----------|----------|
| ❌ `INVALID_PURCHASING` 等复合状态 | "生命周期"与"执行状态"混合 |
| ❌ 所有状态压扁成一个枚举 | 无法表达并行维度 |

**输出要求**：
- 明确写 `(维度名)_status`
- 附带"跨维度不变式"圈定合法区域

---

### 法则2：属性事实与计算分离

**核心法则**：实体只存储"发生过的事实"，绝不存储可以计算出来的结果。

**理论背景**：Codd 关系范式（3NF/BCNF）——消除传递依赖与冗余。

| 禁止事项 | 禁止原因 |
|----------|----------|
| ❌ 计算列：`total_price`、`discounted_amount` | 规则引擎运行时计算，不是实体固有数据 |
| ❌ 状态附属属性：只在某状态有意义的属性 | 应剥离成独立实体（如 Shipment） |

**输出要求**：
- 属性列表只出现名词性的"不可变事实"
- 计算值写在规则章节

---

### 法则3：规则纯函数与无副作用

**核心法则**：规则必须是纯函数——给定相同输入，必定相同输出，且不改变任何状态。

**理论背景**：决策表（Decision Table，1970s BPM）——代替IF/ELSE伪代码。

| 禁止事项 | 禁止原因 |
|----------|----------|
| ❌ 规则描述包含"修改状态"、"删除"、"发送消息" | 规则只返回布尔值或计算值，动作属于状态机或事件 |
| ❌ "如果满足A，就变成B" | "满足A"是规则，"变成B"是状态机触发器 |

**输出要求**：
- 格式：`WHEN (输入条件) -> THEN (输出值/布尔值)`
- 动作词剥离到状态流转或领域事件

---

### 法则4：事件过去时与最小完备

**核心法则**：事件是"已发生的历史事实"，是对外广播的只读数据。

**理论背景**：DDD 领域事件——跨上下文通信契约。

| 禁止事项 | 禁止原因 |
|----------|----------|
| ❌ 命令式命名：`CancelOrder` | 这是命令，不是事件 |
| ❌ 携带大对象：整个订单 JSON | 污染消费者上下文，破坏最小完备 |

**输出要求**：
- 命名：`{动词}{名词}d`（过去时）
- 携带：最小数据集（如 `orderId, reason`）
- 明确：预期消费者 + 执行动作

---

### 法则5：流程编排与协同分离

**核心法则**：区分"单实体内部编排"和"多实体跨界协同"。

**理论背景**：DDD 聚合边界——由"必须同时保持一致性的不变式"决定。

| 禁止事项 | 禁止原因 |
|----------|----------|
| ❌ 单实体MD写"通知库存扣减、通知积分增加" | 跨界协同应以领域事件形式存在 |
| ❌ 跨界动作写在状态流转表 | 状态机只管内部编排 |

**输出要求**：
- 单实体MD只写编排（状态机）
- 跨界以"领域事件"作为接口契约

## 二、检查层：问题清单

### 状态设计检查

| 检查项 | 检查问题 | 判断 |
|--------|----------|------|
| Q1-正交试错 | 能用"并且"连接两个状态描述当前情况吗？（如"已生效 并且 采购中"） | 若能 → 必须拆分为正交字段 |
| Q2-非法态排除 | 两个维度的交叉点，有没有"听起来像胡话/违法"的象限？（如"未生效 并且 已采购"） | 若有 → 拆分后用不变式封死 |
| Q3-独立演变 | A状态走到下一步，是否强制要求B状态也跟着走？ | 若不强制 → 正交 |

---

### 属性设计检查

| 检查项 | 检查问题 | 判断 |
|--------|----------|------|
| Q1-主键依赖 | 这个属性的值，是由这个实体的 ID 决定的吗？ | 若依赖其他实体 → 违反2NF，移除 |
| Q2-传递依赖 | 如果改了A属性的值，B属性的值是否必然跟着变？ | 若是 → 传递依赖，移除 |
| Q3-可计算性 | 这个属性，是否可通过其他属性加减乘除算出来？ | 若是 → 计算列，移除 |
| Q4-身份认同 | 如果两个实体的这组属性完全一样，它们是同一个东西吗？ | 若是 → 应抽成值对象 |

---

### 规则设计检查

| 检查项 | 检查问题 | 判断 |
|--------|----------|------|
| Q1-完备性 | 有无条件组合无动作的"灰度地带"？ | 若有 → 找产品确认 |
| Q2-冲突性 | 有无相同条件不同动作？ | 若有 → 规则矛盾，重构 |
| Q3-纯度 | 规则是否返回布尔值或计算值，而非执行动作？ | 若含动作 → 剥离到状态机/事件 |

---

### 事件设计检查

| 检查项 | 检查问题 | 判断 |
|--------|----------|------|
| Q1-命名 | 命名是否过去时？（`OrderCancelled` ✓，`CancelOrder` ✗） | 若命令式 → 改为过去时 |
| Q2-最小完备 | payload 是否只放消费者处理此事的最小数据集？ | 若过大 → 精简 |
| Q3-消费者 | 是否写明"预期消费者"？反向校验数据完备性 | 若缺失 → 补充消费者 |

---

### 边界设计检查

| 检查项 | 检查问题 | 判断 |
|--------|----------|------|
| Q1-事务一致性 | 当A变更时，B是否必须在这一毫秒内同步变更？ | 若否 → 拆分为独立模型 |
| Q2-不变式保护 | 如果两个实体的数据不一致，系统是否立刻崩溃？ | 若只是最终一致 → 拆分 |
| Q3-并发修改者 | 这两个实体是否会被不同人/请求同时修改？ | 若是 → 必须拆分，避免死锁 |

## 三、自检清单：输出前强制执行

**触发时机**：`/m-design` Step 4 输出草稿前，大模型必须静默执行以下自检。

| 序号 | 检查项 | 检查句式 | 测试示例 |
|------|--------|----------|----------|
| 1 | `[状态正交]` | 检查所有 status 字段，是否存在可以用"并且"连接的双状态？若有，拆分为正交维度，并补充跨维度不变式。 | 测试："正在X并且Y"是否有意义？如"正在执行 并且 部分入库"有意义 → 入库进度应为独立维度 |
| 2 | `[属性除冗]` | 检查实体属性，是否存在可通过其他字段计算得出的值？若有，移除并转为规则。检查是否存在只在某状态有意义的属性？若有，剥离成独立实体。 | 测试：改了A，B是否必然跟着变？若是 → 移除B |
| 3 | `[规则纯度]` | 检查所有规则描述，是否包含"修改状态"、"删除"、"发送"等动作词？若有，将动作剥离到状态流转或领域事件，规则只保留布尔判断或计算返回。 | 测试：规则中是否有"修改"、"发送"、"删除"等动词？若有 → 剥离到事件 |
| 4 | `[事件过去时]` | 检查所有事件命名，是否命令式而非过去时？若有，改为 `{Verb}{Noun}d` 格式。检查 payload 是否过大？若有，精简为最小完备数据集，并写明预期消费者。 | 测试：事件名是否为 `XxxD` 过去时？若为 `Xxx` 命令式 → 改为过去时 |
| 5 | `[边界隔离]` | 检查本实体MD，是否存在跨界协同动作（如"通知库存扣减"）？若有，以领域事件形式移到事件章节。检查是否存在强依赖其他实体瞬间状态的规则？若否，考虑拆分。 | 测试：是否写了"通知X"、"调用Y"？若有 → 剥离为事件 |

---

### 状态正交检查详述

**强制执行**：对每个状态字段，必须填写以下测试表：

| 测试句式 | 是否有意义 | 结论 |
|----------|------------|------|
| "正在{状态A} 并且 {状态B}" | 有意义 / 无意义 | 若有意义 → 拆分为正交维度 |
| "正在{状态A} 并且 {状态C}" | 有意义 / 无意义 | 若有意义 → 拆分为正交维度 |
| ... | ... | ... |

**示例**：
- 测试："正在执行 并且 部分入库" → 有意义 → 入库进度应为独立维度
- 测试："已入库 并且 未结算" → 有意义 → 结算进度应为独立维度
- 测试："草稿 并且 已入库" → 无意义（胡话） → 跨维度不变式封死

## 四、校验示例

### 状态正交示例

**错误设计**：
```
status: DRAFT | PENDING | PURCHASING | INVALID_PURCHASING | COMPLETED
```

**问题**：`INVALID_PURCHASING` 混合了"生效维度"和"采购维度"。

**正确设计**：
```
effective_status: ACTIVE | INVALID
purchase_status: 未采购 | 部分采购 | 已采购

不变式：
- effective_status = INVALID 且 purchase_status ≠ 未采购 → 禁止删除
- effective_status = INVALID → 禁止采购操作
```

---

### 属性除冗示例

**错误设计**：
```
Order 实体属性：
- total_amount (订单总金额)
- discounted_amount (折后金额)
```

**问题**：`total_amount` 可通过明细累加计算，`discounted_amount` 可通过规则计算。

**正确设计**：
```
Order 实体属性：
- (移除 total_amount)
- (移除 discounted_amount)

规则章节：
- R01: WHEN 所有明细数量×单价 → THEN 订单总金额
- R02: WHEN VIP客户且金额>100 → THEN 折后金额 = 总金额 × 0.9
```

---

### 规则纯度示例

**错误设计**：
```
R01: 如果库存不足，修改订单状态为待采购，并通知采购部门
```

**问题**：规则包含"修改状态"和"通知"动作。

**正确设计**：
```
规则章节：
- R01: WHEN 库存数量 < 需求数量 → THEN return False (校验失败)

状态流转章节：
- 当前状态 = CONFIRMED，触发条件 = R01返回False → 目标状态 = PENDING_PURCHASE

领域事件章节：
- StockInsufficientDetected → 携带 orderId, shortage → 消费者：采购模块 → 执行动作：创建采购申请
```

## 六、规约优先级

| 优先级 | 规约类型 | 文件 |
|--------|----------|------|
| 1（最高） | 用户红线（用户原话） | draft/{domain}.md |
| 2 | 建模元规则 | config/meta-design-rules.md |
| 3 | DDD 建模规约 | config/rules.md |
| 4 | 领域规约 | design/domain/{domain}.md |

**说明**：建模元规则（本文件）优先级高于 DDD 建模规约（rules.md），因为 meta-design-rules 定义的是"设计基因"，rules.md 定义的是"业务约束"。
