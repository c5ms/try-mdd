# MDD 设计推演

本 skill **不自动触发**，需用户通过 `/m-design` command 调用。

---

## 核心理念

```
规约先行 → 需求明确 → 角色协作 → 方案输出 → 用户确认 → 设计更新
```

---

## 输入

- 领域名称：由 command 传入
- 业务描述：用户提供的业务需求描述

---

## 步骤

### Step 0：读取已有设计和待确认方案

**目的**：结合现有设计进行推演，避免冲突，发现关联。

**读取顺序**：
1. `design/model.md` —— 领域模型汇总（统一术语表、聚合间引用）
2. `design/todo.md` —— 设计待定问题（待确认、已解决问题）
3. `design/domain/*.md` —— 已设计的领域模型（平铺文件）
4. `solution/*.md` —— 待确认方案（可能关联当前设计）

**信息提取**：

| 来源 | 提取内容 | 用途 |
|------|----------|------|
| design/model.md | 统一术语表 | 已定义术语，避免重复定义或术语冲突 |
| design/model.md | 聚合间引用 | 已有聚合边界，确保新设计引用方式一致 |
| design/model.md | 领域概览 | 已设计领域状态，发现关联领域 |
| design/domain/*.md | 领域模型内容 | 已设计领域的详细模型 |
| design/todo.md | 待确认问题 | 影响当前设计的未确认事项 |
| design/todo.md | 已解决问题 | 已有解决方案可参考 |
| solution/*.md | 方案内容 | 待确认方案中是否有相关设计思路可参考 |

**关联判断**：
- 术语关联：当前设计是否使用已定义术语
- 聚合关联：当前设计是否引用已有聚合（如订单 → 客户）
- 待定项关联：当前设计是否能解决已有待确认问题

**读取后输出**（在推演开始时告知用户）：
```
已有设计关联分析：

已定义术语：
- {术语1}（来自 {领域}）
- {术语2}（来自 {领域}）

聚合间引用：
- {聚合A} → {聚合B}（已有引用关系）

待确认问题：
- {问题1}（来自 {领域}，可能影响当前设计）

已解决问题：
- {问题1}（解决方案：{方案}，可参考）

待确认方案：
- {方案文件}（可能关联当前设计）
```

---

### Step 1：读取设计原则

读取 `config/rules.md`，明确 DDD 建模规约：

**设计原则**：
- DDD 建模语言（实体、值对象、聚合、领域事件）
- 领域边界划分（限界上下文、聚合边界）
- 统一语言定义（术语表）
- 建模设计约束（聚合间引用、一致性边界）

---

### Step 2：确定角色视角

读取 `config/roles.md`，确定可参与设计的角色。

**角色信息提取**：
- 角色名：roles.md 中定义的角色
- 视角：该角色的专业视角
- 关注点：该角色评审时的关注点

**评审维度**：根据 roles.md 中每个角色的「关注点」动态确定

---

### Step 3：明确需求

#### 3.1 理解整理

将用户的自然语言输入理解整理，提取：
- 核心实体
- 关联关系
- 状态定义
- 业务规则
- 用例场景

**用户原话保留**：
> "{用户原始描述}"

#### 3.2 不确定项处理

**如果 AI 不确定**：使用 **AskUserQuestion** 询问用户
**如果用户不确定**：记录到「待定任务清单」

#### 3.3 选定参与角色

根据业务场景和 `config/roles.md` 中定义的角色，选定参与设计的角色：

**选定规则**：
- 根据场景特征匹配角色的「视角」属性
- 涉金额/合规场景 → 选择有合规审计视角的角色
- 涉业务流程场景 → 选择有业务流程视角的角色
- 不确定时使用 **AskUserQuestion** 询问用户

---

### Step 4：角色协作评审

#### 4.1 AI 分析方案

根据专业知识分析解决方案（使用内置模板）：

```markdown
## AI 分析方案

### 术语表
- {从需求提取的术语}

### 实体设计
- {实体列表及属性}

### 聚合边界
- {聚合划分}

### 状态流转
- {状态机设计}

### 业务规则
- {规则清单}

### 数据流图
- {数据流转设计}

### 业务场景
- {场景列表}
```

#### 4.2 逐角色评审

每个角色从自己的视角分析方案，提出意见：

**评审格式**：

```markdown
## 角色：{角色名}

### 评审视角
{该角色的关注点}

### 方案分析
| 维度 | 评价 | 意见 |
|------|------|------|
| {维度1} | ✓/✗/△ | {具体意见} |

### 补充建议
- {该角色提出的补充建议}

### 风险提示
- {该角色发现的风险点}
```

**评审维度**：根据 roles.md 中该角色的「关注点」属性确定

---

### Step 5：输出方案到 solution

将完整方案输出到 `solution/{domain}.md`。

**新增待定项到 `design/todo.md`**：

如果方案中有待定项，追加到 `design/todo.md` 的"一、待确认问题"章节：

```markdown
| {序号} | {问题描述} | {所属领域} | {来源} | {备注} |
```

**方案文件结构**：

```markdown
# 设计方案

## 一、需求概述
{整理后的需求摘要}

## 二、用户原话
> "{用户原始描述}"

## 三、AI 分析方案
{AI 提出的设计方案}

## 四、角色评审汇总
{各角色的评审意见汇总}

## 五、综合方案
{综合各方意见后的最终方案}

## 六、待定任务
{未确定的任务清单}

## 七、设计产出预览
{即将输出的领域模型文件预览}
```

---

### Step 6：询问用户确认

使用 **AskUserQuestion** 询问用户：

```
方案已输出到 solution/{domain}.md

是否将方案更新到领域模型目录 design/domain/{domain}.md？
```

**选项**：
- 确认更新 —— 执行 Step 7
- 暂不更新 —— 方案保留在 solution 目录
- 修改方案 —— 根据用户意见调整方案

---

### Step 7：更新设计

用户确认后，将方案更新到 `design/domain/{domain}.md`。

**大模型判断是否拆分补充文件**：

| 补充类型 | 判断维度 | 拆分决策 |
|----------|----------|----------|
| 状态机 | 状态数量、流转复杂度、异常分支、业务重要性 | 复杂 → `{domain}-state.md`，简单 → 合并在主文件 |
| 业务规则 | 规则数量、规则逻辑复杂度、跨领域关联 | 复杂 → `{domain}-rule.md`，简单 → 合并在主文件 |
| 数据流图 | 系统交互数量、数据流分支、外部集成 | 复杂 → `{domain}-flow.md`，简单 → 合并在主文件 |
| 业务场景 | 场景数量、分支流程、异常处理 | 复杂 → `{domain}-scene.md`，简单 → 合并在主文件 |

**拆分标准**：大模型根据复杂度判断，不做量化指标定义。

**更新 `design/todo.md`**：

如果有问题在设计过程中已解决，将其从"待确认问题"移到"已解决问题"：

```markdown
| {序号} | {问题} | {所属领域} | {解决方案} | {解决日期} |
```

**使用内置模板**（见下方）

## 约束

- **基于已有设计** —— 推演前必须先读取 design/model.md、design/todo.md、design/domain/*.md 和 solution/*.md
- **规约遵循**：推演必须遵循 `config/rules.md` 中的 DDD 建模规约
- **角色必选**：必须选定至少一个角色参与评审
- **待定记录**：不确定项必须追加到 design/todo.md
- **待定状态流转**：已解决问题从"待确认"移到"已解决"
- **用户确认**：更新设计前必须经用户确认
- **聚合间引用**：通过 ID 引用，不直接持有对象
- **平铺结构**：领域模型文件平铺在 design/domain/ 目录
- **拆分判断**：大模型根据复杂度判断是否拆分补充文件，不做量化指标定义
