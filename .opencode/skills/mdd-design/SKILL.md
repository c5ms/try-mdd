# MDD 设计推演

本 skill **不自动触发**，需用户通过 `/m-design` command 调用。

---

## 核心理念

```
调研 → 设计 → 输出（自由形式）
```

---

## 输入

- 领域名称：由 command 传入
- 业务描述：用户提供的业务需求描述

---

## 输出要求

必须覆盖以下内容（形式不限）：

| 章节 | 必须覆盖 | 说明 |
|------|----------|------|
| 领域词典 | ✓ | 概念定义表（表格、列表均可） |
| 状态流转 | ✓ | 状态矩阵、状态图（mermaid 或其他形式） |
| 实体定义 | ✓ | ER图、属性表（表格或其他形式） |
| 业务规则 | ✓ | 规则表、不变式（表格或其他形式） |
| 领域事件 | ✓ | 事件表（表格或其他形式） |
| 聚合边界 | ✓ | 聚合表（表格或其他形式） |
| 上下游关系图 | ✓ | mermaid graph 或其他形式 |
| 用例 | ✓ | 用例表（表格或其他形式） |
| 补充流程图 | △ | 可选，如需要可补充 |
| 待定任务 | ✓ | 待定项表（表格或其他形式） |

**用户自定义内容**：
- 可以追加任何用户希望的内容
- 形式不限
- 在 mdd-apply 时会被整理到"附录"章节

**硬约束**：
- 用户原话必须保留（不可省略）

---

## 步骤

### Step 0：明确输出目标

读取 `mdd-apply/references/domain-model-template.md`，了解需要覆盖的内容。

**目的**：
- 知道最终模型需要哪些章节
- 但不强制遵循模板格式
- 设计时自由探索，mdd-apply 会整理成标准格式

---

### Step 1：调研已有设计

**目的**：结合现有设计进行推演，避免冲突，发现关联。

**读取顺序**：
1. `config/rules.md` —— DDD 建模规约
2. `config/roles.md` —— 角色定义
3. `design/model.md` —— 领域模型汇总
4. `design/todo.md` —— 设计待定问题
5. `design/domain/*.md` —— 已设计的领域模型
6. `draft/*.md` —— 待确认草稿

**信息提取**：

| 来源 | 提取内容 | 用途 |
|------|----------|------|
| config/rules.md | DDD建模规约 | 确保聚合边界、引用方式正确 |
| config/roles.md | 角色视角 | 选择角色参与评审 |
| design/model.md | 聚合间引用 | 确保新设计引用方式一致 |
| design/domain/*.md | 领域模型内容 | 发现关联领域、术语冲突 |
| design/todo.md | 待确认问题 | 影响当前设计的未确认事项 |
| draft/*.md | 草稿内容 | 相关设计思路可参考 |

---

### Step 2：明确需求

#### 2.1 理解整理

将用户的自然语言输入理解整理，提取：
- 核心实体
- 关联关系
- 状态定义
- 业务规则（仅用户明确表达的）
- 用例场景

**用户原话保留**：
> "{用户原始描述}"

#### 2.2 不确定项处理

| 场景 | 行动 |
|------|------|
| 推演不确定 | AskUserQuestion 询问用户 |
| 用户不确定 | 记录到待定任务清单 |
| 用户未提及但认为必要 | AskUserQuestion 让用户澄清 |
| 用户未提及且靠经验认为必要 | 禁止设计 |

---

### Step 3：角色协作评审

#### 3.1 模型设计方案

根据用户输入分析解决方案。

#### 3.2 逐角色评审

每个角色从自己的视角分析方案，提出意见。

---

### Step 4：输出草稿

将设计内容输出到 `draft/{domain}.md`。

**输出内容**：
- 必须覆盖 domain-model-template 的10章节内容（形式自由）
- 用户原话必须保留
- 可以追加用户自定义内容
- 待定项追加到 `design/todo.md`

---

## 输出

完成后输出：
```
设计推演完成：

草稿文件：
- draft/{domain}.md

提示：
- 使用 /m-apply 将草稿应用到设计目录
- mdd-apply 会自动整理成标准格式
```

## 设计约束（MVP + 惰性原则）

| 约束 | 说明 |
|------|------|
| **MVP原则** | 只设计"如果没有的话绝对不可能"的属性 |
| **惰性设计原则** | 属性设计是惰性的，除非用户要求才会设计其他属性 |
| **禁止靠经验设计业务规则** | 只输出MVP规则，除非用户明确表达 |
| **禁止靠经验猜测** | 不确定时 AskUserQuestion 让用户澄清 |

**判断标准**：

| 场景 | 判断 | 行动 |
|------|------|------|
| 用户明确要求某属性 | ✓ 设计 | 直接设计 |
| 用户给出属性列表 | ✓ 设计 | 按用户列表设计 |
| 用户未提及，但认为必要 | △ 不确定 | AskUserQuestion 让用户澄清 |
| 用户未提及，靠经验认为必要 | ✗ 禁止 | 不设计 |

---

## 禁止内容（防污染原则）

| 内容 | 禁止原因 |
|------|----------|
| 技术实现 | 大模型推演业务不需要知道技术实现 |
| UI/UX描述 | 这是原型的活，不是模型的活 |
| 非结构化背景故事 | 废话，大模型不需要 |
| 性能要求 | 这是架构文档的事 |
| 值对象章节 | 过度DDD抽象，对推演价值低 |
| 默认术语表 | 术语表应是用户澄清才记录 |

---

## 约束

| 约束 | 说明 |
|------|------|
| **用户原话保留** | 草稿中必须包含用户原话（硬约束） |
| **覆盖模板内容** | 必须覆盖 domain-model-template 的10章节内容（形式不限） |
| **基于已有设计** | 推演前必须调研 config/rules.md、design/model.md、design/domain/*.md、draft/*.md |
| **规约遵循** | 推演必须遵循 DDD 建模规约（聚合间通过ID引用） |
| **角色必选** | 必须选定至少一个角色参与评审 |
| **待定记录** | 不确定项必须追加到 design/todo.md |
| **MVP原则** | 只设计"如果没有的话绝对不可能"的属性 |
| **惰性设计** | 属性设计是惰性的，除非用户要求才会设计其他属性 |
| **禁止靠经验** | 禁止靠经验设计属性、业务规则，禁止靠经验猜测 |
