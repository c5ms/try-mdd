# MDD 设计推演

本 skill **不自动触发**，需用户通过 `/m-design` command 调用。

---

## 核心理念

```
规约先行 → 需求明确 → 角色协作 → 方案输出 → 用户确认 → 设计更新
```

---

## 输入

- 领域名称：由 command 传入
- 业务描述：用户提供的业务需求描述

---

## 步骤

### Step 0：先看模板

读取 `references/domain-model-template.md`，明确输出目标结构（10章节）。

**模板定位：**
- MD文档是给大模型当"运行时内存"用的，而非给人看的文档
- 必须消除歧义、高度结构化、逻辑严密、没有废话
- 剥离所有"实现细节"，只留最纯粹的业务逻辑骨架

**禁止内容（防污染原则）：**

| 内容 | 禁止原因 |
|------|----------|
| 技术实现 | 大模型推演业务不需要知道技术实现 |
| UI/UX描述 | 这是原型的活，不是模型的活 |
| 非结构化背景故事 | 废话，大模型不需要 |
| 性能要求 | 这是架构文档的事 |
| 值对象章节 | 过度DDD抽象，对推演价值低 |
| 默认术语表 | 术语表应是用户澄清才记录 |

---

### 设计约束（MVP + 惰性原则）

| 约束 | 说明 |
|------|------|
| **MVP原则** | 任何模型在最开始的设计，只设计"如果没有的话绝对不可能"的属性 |
| **惰性设计原则** | 属性设计是惰性的，除非用户要求或给出属性列表，才会设计其他属性 |
| **禁止靠经验设计业务规则** | 绝对不允许靠经验设计任何业务规则，默认只输出MVP规则，除非用户表达出了某些规则 |
| **禁止靠经验猜测** | 任何地方不要靠经验猜测，如果认为有必要但不确定，与用户确认让用户澄清 |

**判断标准：**

| 场景 | 判断 | 行动 |
|------|------|------|
| 用户明确要求某属性 | ✓ 设计 | 直接设计 |
| 用户给出属性列表 | ✓ 设计 | 按用户列表设计 |
| 用户未提及，但认为必要 | △ 不确定 | AskUserQuestion 让用户澄清 |
| 用户未提及，靠经验认为必要 | ✗ 禁止 | 不设计 |

**带着目标去调查分析：**
- 调查时时刻对照模板，确保输出结构符合模板要求
- 分析时时刻对照禁止内容表，避免输出污染
- 设计时时刻对照设计约束，禁止靠经验猜测

---

### Step 1：读取已有设计和待确认方案

**目的**：结合现有设计进行推演，避免冲突，发现关联。

**读取顺序**：
1. `design/model.md` —— 领域模型汇总
2. `design/todo.md` —— 设计待定问题
3. `design/domain/*.md` —— 已设计的领域模型
4. `solution/*.md` —— 待确认方案

**信息提取**：

| 来源 | 提取内容 | 用途 |
|------|----------|------|
| design/model.md | 聚合间引用 | 确保新设计引用方式一致 |
| design/domain/*.md | 领域模型内容 | 发现关联领域 |
| design/todo.md | 待确认问题 | 影响当前设计的未确认事项 |
| solution/*.md | 方案内容 | 相关设计思路可参考 |

**关联判断**：
- 术语关联：当前设计是否使用已定义术语
- 聚合关联：当前设计是否引用已有聚合

---

### Step 2：读取设计原则

读取 `config/rules.md`，明确 DDD 建模规约。

---

### Step 3：确定角色视角

读取 `config/roles.md`，确定可参与设计的角色。

---

### Step 4：明确需求

#### 4.1 理解整理

将用户的自然语言输入理解整理，提取：
- 核心实体
- 关联关系
- 状态定义
- 业务规则（仅用户明确表达的）
- 用例场景

**用户原话保留**：
> "{用户原始描述}"

#### 4.2 不确定项处理

**如果推演不确定**：使用 **AskUserQuestion** 询问用户
**如果用户不确定**：记录到「待定任务清单」

---

### Step 5：角色协作评审

#### 5.1 模型设计方案

根据用户输入分析解决方案（使用模板）。

#### 5.2 逐角色评审

每个角色从自己的视角分析方案，提出意见。

---

### Step 6：输出方案到 solution

将完整方案输出到 `solution/{domain}.md`。

**新增待定项到 `design/todo.md`**：

如果方案中有待定项，追加到 `design/todo.md`。

---

### Step 7：询问用户确认

使用 **AskUserQuestion** 询问用户：

```
方案已输出到 solution/{domain}.md

是否将方案更新到领域模型目录 design/domain/{domain}.md？
```

---

### Step 8：更新设计

用户确认后，将方案更新到 `design/domain/{domain}.md`。

**使用模板**：`references/domain-model-template.md`

---

## 输出

完成后输出：
```
设计推演完成：

方案文件：
- solution/{domain}.md

设计文件（如已确认）：
- design/domain/{domain}.md

汇总文件：
- design/model.md（已更新）
- design/todo.md（已更新）
```

## 约束

| 约束 | 说明 |
|------|------|
| **先看模板** | 推演前必须先读取 `references/domain-model-template.md` |
| **基于已有设计** | 推演前必须先读取 design/model.md、design/todo.md、design/domain/*.md |
| **规约遵循** | 推演必须遵循 `config/rules.md` |
| **角色必选** | 必须选定至少一个角色参与评审 |
| **用户原话保留** | 方案中必须包含用户原话章节 |
| **待定记录** | 不确定项必须追加到 design/todo.md |
| **用户确认** | 更新设计前必须经用户确认 |
| **聚合间引用** | 通过 ID 引用，不直接持有对象 |
| **平铺结构** | 领域模型文件平铺在 design/domain/ 目录 |
| **MVP原则** | 只设计"如果没有的话绝对不可能"的属性 |
| **惰性设计** | 属性设计是惰性的，除非用户要求才会设计其他属性 |
| **禁止靠经验** | 禁止靠经验设计属性、业务规则，禁止靠经验猜测 |
