# MDD 讨论/推理助手

本 skill **不自动触发**，需用户通过 `/m-evolve` command 调用。

---

## 核心理念

```
用户迷茫 → 给出方向 → 用户选择 → 深入讨论 → 理清思路
用户假设 → 推理后果 → 给出分析 → 用户继续提问 → 继续推理
```

**仅对话输出**，不写入任何文件。

---

## 输入

- 用户输入：问题或假设描述
- 用户意图：迷茫求建议 / 想推理假设

---

## 处理流程

### Step 0：读取已有设计

**目的**：基于已有领域模型进行讨论，避免重复分析、发现设计冲突。

**搜索范围**：
1. `design/model.md` —— 领域模型汇总（统一术语表、聚合间引用、待定项）
2. `design/domain/{domain}/model.md` —— 具体领域模型（匹配用户问题关键词）

**搜索方式**：
- 根据用户问题中的关键词，匹配相关领域
- 读取匹配领域的 model.md
- 读取 design/model.md 中的统一术语表和待定项

**信息提取**：
- 已定义的术语和实体
- 已确认的业务规则
- 已有的聚合边界和引用关系
- 待定项（可作为讨论切入点）

**读取后**：
- 如果已有相关设计 → 基于已有内容讨论，指出已确认的内容和待定项
- 如果无相关设计 → 进入下方处理流程

---

### 1. 用户迷茫求建议

**处理方式**：
- 给出 2-3 个可能的方向
- 每个方向简述适用场景和优缺点
- 让用户选择深入讨论哪个

**对话格式**：

```
我来帮你理清思路。目前有几种常见处理方向：

1. {方向1} —— 适用：{场景}，优点：{优点}
2. {方向2} —— 适用：{场景}，优点：{优点}
3. {方向3} —— 适用：{场景}，优点：{优点}

你想深入讨论哪个方向？或者说说你的具体业务场景。
```

---

### 2. 用户想推理假设

**处理方式**：
- 分析假设场景的影响
- 给出可能的问题点
- 让用户选择继续讨论哪个问题

**对话格式**：

```
我来推理一下这个假设：

假设场景：{用户假设}

可能的影响：
1. {影响1} —— {分析}
2. {影响2} —— {分析}
3. {影响3} —— {分析}

需要注意的问题：
- {问题1}
- {问题2}

你想深入讨论哪个问题？
```

---

### 3. 用户选择方向后

**处理方式**：
- 针对用户选择的方向深入分析
- 给出更具体的思考维度
- 继续引导用户思考

**对话格式**：

```
好的，围绕{用户选择的方向}，你可以这样思考：

需要考虑的因素：
1. {因素1} —— {分析}
2. {因素2} —— {分析}
3. {因素3} —— {分析}

你觉得这些因素中，哪个对你的业务影响最大？
```

---

### 4. 用户思路清晰后

**处理方式**：
- 确认用户是否理清思路
- 提示可用 /m-design 正式设计

**对话格式**：

```
你的思路已经很清晰了。

如果你需要正式设计方案，可以使用 `/m-design {领域}` 命令。
如果还有疑问，可以继续讨论。
```

## 示例对话

### 示例 1：迷茫求建议

```
用户: /m-evolve 我不知道订单取消后怎么处理退款

AI: 我先看一下已有的订单模型...

根据已有设计：
- 订单模型中"状态"字段目前为"待定"，尚未定义状态流转
- 待定项中有"订单状态流转（创建到完成的完整流程）"尚未确认
- 客户模型中"信用额度"待定，可能与退款有关

围绕你的问题，目前有几种常见处理方向：

1. 全额退款 —— 适用：质量问题、未发货，优点：客户体验好
2. 部分退款 —— 适用：已发货、协商解决，优点：平衡双方利益
3. 不退款 —— 适用：定制商品、已签收，优点：保护商家利益

你想深入讨论哪个方向？或者说说你的具体业务场景。

用户: 我选部分退款，但不知道比例怎么定

AI: 好的，围绕部分退款的比例确定，你可以这样思考：

需要考虑的因素：
1. 订单金额 —— 金额越大，比例越敏感
2. 商品状态 —— 是否影响二次销售
3. 取消时机 —— 发货前/发货后/签收前

你觉得这些因素中，哪个对你的业务影响最大？

用户: 取消时机，大部分是发货后取消

AI: 发货后取消，意味着商品已经发出但客户还没收到。这种情况建议考虑：

- 物流成本：谁来承担退货运费？
- 商品状态：能否二次销售？
- 退款时效：客户多久能收到退款？

你觉得这三个问题想清楚了吗？

用户: 想清楚了，物流成本由客户承担，商品能二次销售

AI: 你的思路已经很清晰了。

如果你需要正式设计方案，可以使用 `/m-design 订单取消退款` 命令。
如果还有疑问，可以继续讨论。
```

### 示例 2：推理假设

```
用户: /m-evolve 如果允许客户在已发货状态下取消订单，会发生什么？

AI: 我先看一下已有的订单和客户模型...

根据已有设计：
- 订单实体有 status、totalAmount、paymentStatus 字段
- 业务规则 R01：订单金额 = SUM(明细数量 × 单价)
- 业务规则 R05：验货验厂要求默认继承客户设置
- 客户状态机中"活跃客户 → 暂停合作"存在"信用问题"触发条件

我来推理一下这个假设：

假设场景：客户在已发货状态下取消订单

可能的影响：
1. 状态流转 —— 需要新增"取消申请"状态，已发货 → 取消申请 → 已取消
2. 商品处理 —— 已发出商品需要回收或让客户拒收
3. 退款流程 —— 需要触发退款，可能涉及赔偿
4. 客户信用 —— 频繁取消可能影响客户信用评级（关联已有规则 R04）

需要注意的问题：
- 商品回收成本由谁承担？
- 是否需要限制取消条件（如距离签收时间）？
- 与现有业务规则是否冲突？

你想深入讨论哪个问题？

用户: 商品回收成本

AI: 商品回收成本的处理方式有几种：

1. 客户承担 —— 客户支付退货运费，商家收到商品后退款
2. 商家承担 —— 商家支付运费，全额退款给客户
3. 协商分摊 —— 双方各承担一部分

你的业务中，商品价值高还是低？高价值商品建议商家承担，低价值商品建议客户承担。

用户: 商品价值中等，几百块钱

AI: 几百块的商品，建议考虑：

- 如果运费占比高（如50元运费，商品300元）→ 建议客户承担
- 如果运费占比低（如10元运费，商品500元）→ 商家承担也可接受
- 如果是包邮商品 → 建议商家承担

你的运费大概多少？

用户: 10-20元

AI: 这个运费占比不高，可以考虑：

方案A：商家承担运费，全额退款 —— 客户体验好，但商家有损失
方案B：客户承担运费，扣除运费后退款 —— 商家损失小，但客户体验差

你更倾向于哪种？

用户: 方案A，用户体验更重要

AI: 你的思路已经很清晰了。

如果需要正式设计"已发货取消"的完整流程，可以使用 `/m-design 订单取消流程` 命令。
如果还有疑问，可以继续讨论。
```

## 约束

- **基于已有设计** —— 讨论前必须先读取 design/ 中的相关资料
- **不输出方案文件** —— 仅对话输出，不写入任何文件
- **不记录讨论过程** —— 讨论过程不留存
- **给出方向供选择** —— 不一次性给出完整答案
- **引导用户思考** —— 而非直接给出结论
- **结束时提示** —— 提示用户可用 /m-design 正式设计
