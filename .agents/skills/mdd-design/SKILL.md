# MDD 设计推演

本 skill **不自动触发**，需用户通过 `/m-design <领域名称>` command 调用。

---

## 核心理念

从用户自然语言描述中：
1. **提取** 实体、关联、状态、规则、用例
2. **推演** 异常分支、边界条件、漏洞风险
3. **产出** 结构化设计文档

---

## 输入

- 对话记录文件：`inputs/{domain}/input.md`
- 领域名称：由 command 传入

---

## 步骤

### Step 1：读取对话

读取 `inputs/{domain}/input.md`，提取关键信息：

**提取维度**：
| 维度 | 关注内容 |
|------|----------|
| 实体 | 用户提到的业务对象（订单、客户、SKU） |
| 关联 | 实体间的关系（属于、包含、关联） |
| 状态 | 用户提到的状态词（待支付、已发货） |
| 触发 | 状态变化的原因（支付、发货、取消） |
| 规则 | 用户提到的约束（必须、不能、超过） |
| 用例 | 用户描述的业务场景 |

---

### Step 2：角色协作推演

多角色视角补充遗漏场景：

| 角色 | 关注角度 | 输出内容 |
|------|----------|----------|
| 业务分析师 | 用例完整性、边界条件 | 遗漏场景、异常分支 |
| 财务审核员 | 合规性、金额计算 | 涉金额时的风险提示 |
| 测试工程师 | 并发、时序、数据一致性 | 状态死锁、孤儿状态 |

**参与规则**：
- 涉金额 → 财务审核员参与
- 有状态流转 → 测试工程师参与
- 默认 → 业务分析师参与

---

### Step 3：产出写入

使用 **TodoWrite** 跟踪进度，依次写入：

#### 3.1 model.md —— 业务模型

```markdown
## 实体

### {实体名}
| 属性 | 类型 | 说明 |
|------|------|------|
| {属性} | {类型} | {说明} |

## 关联

- {实体A} → {实体B}: {关系}（如：1:N）
```

#### 3.2 states.md —— 状态机

```markdown
## 状态定义

| 状态 | 编码 | 是否终态 |
|------|------|----------|
| {状态} | {CODE} | 是/否 |

## 正常流转

- {状态A} → [{事件}] → {状态B}
  - 前置：{条件}

## 异常流转（推演补充）

- {状态A} → [{异常事件}] → {异常状态}
  - 场景：{并发/越权/边界}
```

#### 3.3 rules.md —— 业务规则

```markdown
## 红线（用户明确约束）

- {规则编号}: {规则描述}
  - 来源：用户原话
  - 违反后果：{后果}

## 推演规则（AI补充）

- {规则编号}: {规则描述}
  - 来源：角色推演
  - 场景：{触发场景}
```

#### 3.4 flows.md —— 用例流程图

```markdown
## {用例名}

```mermaid
graph LR
  A[起点] --> B[步骤1] --> C[步骤2] --> D[终点]
\```

## 异常分支

- {异常描述}
```

---

### Step 4：风险检测

写入完成后，检查以下风险点：

| 风险类型 | 检查方法 |
|----------|----------|
| 状态死锁 | 是否存在无法到达终态的状态 |
| 孤儿状态 | 是否存在无法到达的状态 |
| 红线击穿 | 推演规则是否能绕过用户红线 |
| 并发冲突 | 同一状态多个触发事件的竞态 |

如有风险，追加到 `rules.md` 的「风险提示」章节。

---

## 输出

产出文件：
- `outputs/domain/{domain}/model.md`
- `outputs/domain/{domain}/states.md`
- `outputs/domain/{domain}/rules.md`
- `outputs/domain/{domain}/flows.md`

---

## 约束

- **输入隔离**：只读 `inputs/`，禁止修改
- **输出覆盖**：每次推演完全覆盖 `outputs/`
- **不编造**：未从对话提取的信息标注「待确认」
- **角色参与**：根据业务特征选择性调用角色视角
