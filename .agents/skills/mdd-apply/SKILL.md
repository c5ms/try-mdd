# MDD 方案应用

本 skill **不自动触发**，需用户通过 `/m-apply` command 调用。

---

## 核心理念

```
分析草稿箱 → 整理内容 → 澄清（可选） → 领域划分决策 → 标准化输出 → 更新模型
```

---

## 输入

由 `/m-apply` command 触发，无需额外参数。

---

## 步骤

### Step 0：读取模板

读取以下模板，明确输出目标结构：

1. `.agents/skills/mdd-apply/references/domain-model-template.md` —— 领域模型模板（11章节）
2. `.agents/skills/mdd-apply/references/model-summary-template.md` —— 模型汇总模板（3章节）

---

### Step 1：搜索所有草稿

搜索 `draft/` 目录下所有的草稿文件：

```
draft/{domain}.md
```

**搜索方式**：使用 Glob 工具搜索 `draft/*.md`

**输出格式**：

```
发现待应用草稿：

| # | 领域 | 草稿文件 |
|---|------|----------|
| 1 | {domain1} | draft/{domain1}.md |
| 2 | {domain2} | draft/{domain2}.md |
```

---

### Step 2：分析草稿箱整体

**目的**：发现草稿间的关联、冲突、重叠，判断是否需要重新划分领域。

#### 2.1 读取所有草稿

读取所有草稿文件内容。

#### 2.2 分析草稿间关联

| 分析项 | 检测内容 | 输出 |
|--------|----------|------|
| 术语冲突 | 不同草稿中同一术语的定义是否一致 | 冲突术语列表 |
| 聚合边界重叠 | 不同草稿中是否有相同聚合根或实体 | 重叠聚合列表 |
| 上下游关系一致性 | 草稿间的引用关系是否与现有模型一致 | 不一致列表 |

#### 2.3 判断是否需要重新划分领域

**触发重新划分的情况**：

| 情况 | 判断标准 | 建议方案 |
|------|----------|----------|
| 多个草稿涉及同一聚合根 | 多个草稿中定义了相同聚合根 | 合并为一个领域 |
| 一个草稿涉及多个聚合根 | 一个草稿中定义了多个独立聚合根 | 拆分为多个领域 |
| 草稿领域边界与现有领域冲突 | 草稿的聚合与现有领域的聚合重叠 | 调整领域边界 |
| 术语定义冲突 | 不同草稿对同一术语定义不一致 | 统一术语或拆分领域 |

**输出分析结果**：

```
草稿箱整体分析结果：

| 问题类型 | 涉及草稿 | 建议方案 |
|----------|----------|----------|
| {问题1} | draft/a.md, draft/b.md | {建议方案} |
| {问题2} | draft/c.md | {建议方案} |

领域划分建议：
- {domain1}：合并 draft/a.md + draft/b.md
- {domain2}：拆分 draft/c.md 为 {domain2-1} + {domain2-2}
- {domain3}：调整边界，将 {聚合X} 从 {domainY} 移入
```

---

### Step 3：逐个确认草稿

**逐个确认每个草稿的领域归属和处理方式。**

对每个草稿：

```
草稿 {domain}.md 分析结果：

领域归属建议：
- 原领域：{domain}
- 建议领域：{domain-new}（如需调整）
- 建议原因：{原因}

与现有领域关系：
- 上游依赖：{上游领域列表}
- 下游依赖：{下游领域列表}
- 聚合边界影响：{影响说明}

发现的问题：
- {问题1}
- {问题2}

用户自定义内容：
- {自定义章节1}
- {自定义章节2}

确认应用此草稿？
```

**用户选择**：
- 确认应用
- 需要调整（用户指定调整方案）
- 暂不应用

---

### Step 4：继续澄清（可选）

如果分析中发现不确定项：

```
在分析草稿 {domain}.md 时发现以下不确定项：

- {不确定项1}（来源：{来源说明}）
- {不确定项2}（来源：{来源说明}）

是否需要澄清？
```

**用户选择**：
- 需要澄清 → 使用 AskUserQuestion 逐个澄清
- 不需要澄清 → 记录到待定任务

---

### Step 5：领域划分决策

**如果 Step 2 分析建议重新划分领域，确认领域划分方案。**

```
领域划分方案：

| 操作 | 原领域 | 新领域 | 涉及草稿 |
|------|--------|--------|----------|
| 合并 | {domain1}, {domain2} | {domain-new} | draft/a.md, draft/b.md |
| 拆分 | {domain3} | {domain3-1}, {domain3-2} | draft/c.md |
| 调整 | {domain4} | {domain4}（边界调整） | draft/d.md |

对现有模型的影响：
- 需删除：design/domain/{domain}.md（如合并/拆分）
- 需创建：design/domain/{domain-new}.md
- 需更新：design/model.md（领域全景图、聚合根索引）

确认此划分方案？
```

**用户选择**：
- 确认方案
- 调整方案（用户指定调整）
- 暂不划分（按原领域处理）

---

### Step 6：标准化输出

对用户确认的每个草稿，整理成标准格式。

#### 6.1 解析草稿内容

从草稿中提取：
- 用户原话
- 领域词典
- 状态流转
- 实体定义
- 业务规则
- 领域事件
- 聚合边界
- 上下游关系图
- 用例
- 补充流程图（如有）
- 待定任务
- **用户自定义内容**

#### 6.2 整理成标准格式

使用 `.agents/skills/mdd-apply/references/domain-model-template.md` 生成标准格式：

**标准章节（第1-10章）**：

| 章节 | 内容来源 |
|------|----------|
| 1 领域词典 | 从草稿提取，整理为表格 |
| 2 状态流转 | 从草稿提取，整理为表格 + mermaid 状态图 |
| 3 实体定义 | 从草稿提取，整理为 ER图 + 属性表 |
| 4 业务规则 | 从草稿提取，整理为规则表 + 不变式表 |
| 5 领域事件 | 从草稿提取，整理为事件表 |
| 6 聚合边界 | 从草稿提取，整理为聚合表 |
| 7 上下游关系图 | 从草稿提取，整理为 mermaid graph LR |
| 8 用例 | 从草稿提取，整理为用例表 |
| 9 补充流程图 | 从草稿提取（如有），整理为 mermaid graph |
| 10 待定任务 | 从草稿提取，整理为待定项表 |

**附录章节（第11章）**：

| 章节 | 内容来源 |
|------|----------|
| 11 附录：用户自定义内容 | 从草稿提取用户自定义内容，单独章节 |

#### 6.3 写入领域模型

将生成的模型写入 `design/domain/{domain}.md`

---

### Step 7：清理草稿文件

更新成功后，删除已应用的草稿文件：

- 删除 `draft/{domain}.md`

---

### Step 8：更新汇总文件

自动更新 `design/model.md`（领域模型汇总）。

#### 8.1 读取当前汇总文件

读取 `design/model.md`，提取：
- 领域全景图（mermaid graph TB）
- 聚合根索引表
- 演进记录表

#### 8.2 更新领域全景图

根据新应用的领域，更新全景图：

| 操作 | 说明 |
|------|------|
| 新增领域节点 | 根据新领域添加节点 |
| 新增引用关系 | 根据草稿的"上下游关系图"添加引用 |
| 删除领域节点 | 如合并/拆分领域，删除旧节点 |
| 更新引用关系 | 如调整边界，更新引用箭头 |

#### 8.3 更新聚合根索引

| 操作 | 说明 |
|------|------|
| 追加新聚合根 | 新领域的聚合根 |
| 删除旧聚合根 | 如合并/拆分，删除旧聚合根 |
| 更新聚合根信息 | 如调整边界，更新聚合根所属域 |

#### 8.4 更新演进记录

追加本次应用记录：

| 数据 | 来源 |
|------|------|
| 日期 | 当前日期 |
| 变更内容 | 完成{领域}领域设计 / 合并领域 / 拆分领域 / 调整领域边界 |
| 涉及领域 | 领域名称列表 |

#### 8.5 写入汇总文件

将更新的内容写入 `design/model.md`

---

完成后输出：

```
应用完成：

已更新：
- design/domain/{domain1}.md    —— {领域1} 领域模型（标准格式 + 用户自定义附录）
- design/domain/{domain2}.md    —— {领域2} 领域模型
- design/model.md    —— 领域模型汇总（已更新全景图、聚合根索引、演进记录）

已清理：
- draft/{domain1}.md
- draft/{domain2}.md

领域划分调整（如有）：
- 合并：{domain1} + {domain2} → {domain-new}
- 拆分：{domain3} → {domain3-1} + {domain3-2}
- 调整边界：{domain4} 边界调整

待澄清项（如有）：
- {待定项1} 已记录到 design/todo.md
- {待定项2} 已澄清并更新到模型
```

---

## 约束

| 约束 | 说明 |
|------|------|
| **智能分析** | 必须分析草稿箱整体，发现关联、冲突、重叠 |
| **自动提出** | 发现需要重新划分领域时，主动提出建议 |
| **逐个确认** | 用户逐个确认每个草稿的领域归属和处理方式 |
| **继续澄清** | 如发现不确定项，可继续跟用户确认 |
| **标准化输出** | 草稿内容整理成 domain-model-template 标准格式 |
| **附录处理** | 用户自定义内容整理到第11章"附录" |
| **领域划分考虑** | 可能合并、拆分、调整领域边界 |
| **目录结构调整** | 可能删除、创建、移动 design/domain/ 文件 |
| **失败处理** | 如果某个草稿更新失败，跳过并记录错误，继续处理其他草稿 |
| **清理时机** | 只有更新成功后才删除草稿文件 |
| **模板遵循** | design/domain/{domain}.md 必须遵循 domain-model-template 的11章节结构 |
