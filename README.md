# MDD 设计引擎

> Model-Driven Design Framework - 领域模型先行设计工具

---

## 一、项目定位

### 框架定位

MDD（Model-Driven Design）设计框架，用于：

- **领域模型设计** → 实体、聚合、值对象建模
- **模型论证** → 多角色协作评审验证
- **模型推演** → 状态流转、业务规则、数据流推演
- **方案输出** → 统一术语表、领域关系图

**核心理念**：在业务开发前快速挖掘需求、设计方案，是**原型阶段之前的高度设计工具**。

### 当前项目应用

服装外贸公司ERP系统，支持工贸一体化贸易公司全业务线管理。

---

## 二、为什么要有这个项目

### 与 SDD（Spec-Driven Development）的区别

| 维度 | SDD（OpenSpec/Spec Kit） | MDD（本框架） |
|------|--------------------------|---------------|
| **目标** | 代码落地 | 设计验证 |
| **阶段** | 开发阶段 | 需求挖掘阶段 |
| **产物** | Spec → 代码生成/辅助开发 | 领域模型 → 推演验证 → 方案 |
| **验证时机** | 代码运行后验证 | 推演阶段角色评审验证 |
| **用户** | 开发人员 | 业务人员 + 设计人员 |
| **沟通语言** | 技术规格 | 业务术语 |

**定位差异**：
- **SDD**：帮助开发人员快速落地代码，减少编码工作量
- **MDD**：帮助业务人员与设计人员快速推理需求，验证设计方案，在代码之前完成高度设计

### 产出物

| 产出 | 文件 | 说明 |
|------|------|------|
| 统一术语表 | design/model.md | 业务人员与开发人员共享语言 |
| 领域模型 | design/domain/{domain}/model.md | 实体、聚合、状态机、规则 |
| 领域关系图 | design/model.md | Mermaid 聚合间引用图 |
| 待定项清单 | design/model.md | 未确认事项汇总，便于追溯 |

### 收益

| 收益 | 说明 |
|------|------|
| **降低返工成本** | 在代码编写前完成设计验证，避免需求理解偏差导致重构 |
| **统一沟通语言** | 业务人员与开发人员使用相同术语，减少沟通歧义 |
| **角色协作验证** | 业务经理、财务主管、架构师等多视角评审，减少遗漏场景 |
| **快速迭代** | 推演阶段快速调整方案，无需重构代码 |
| **待定项追溯** | 未确认事项记录在待定清单，避免被遗忘 |

## 三、如何使用

### Commands

| 命令 | 用途 | 示例 |
|------|------|------|
| `/m-init` | 项目初始化 | 确认定位、设计角色、确定准则、初始化目录 |
| `/m-design {领域}` | 领域模型设计推演 | `/m-design 订单管理` |
| `/m-apply` | 批量应用方案到领域模型 | 将 solution/*.md 应用到 design/domain/ |
| `/m-evolve <问题>` | 讨论/推理助手 | `/m-evolve 订单状态应该如何设计？` |
| `/m-reflect` | 反思改进 | 总结设计错误、改进流程 |

### 工作流

```
用户: /m-design 订单业务模型，订单属于一个客户...
  ↓
读取规约 → config/rules.md（DDD 建模规约）
  ↓
确定角色 → config/roles.md（角色视角）
  ↓
明确需求 → 用户输入（AI理解整理）
  ↓
角色评审 → 业务经理、财务主管、架构师逐角色评审
  ↓
输出方案 → solution/{domain}.md（待确认方案）
  ↓
用户确认 → 是否更新到领域模型？
  ↓
更新设计 → design/domain/{domain}/model.md
  ↓
更新汇总 → design/model.md（统一术语表、领域关系图）
```

### 自定义设计规则

设计规则定义在 `config/rules.md`，包含：

| 规则类型 | 说明 |
|------|------|
| 项目定位 | 企业级/互联网，复杂度等级 |
| 设计准则 | 数据一致性、并发处理、权限控制、业务合规等 |
| DDD 建模规约 | 聚合边界、实体/值对象判断、领域事件命名 |

**三种自定义方式**：

| 方式 | 操作 | 适用场景 |
|------|------|----------|
| 直接编辑 | 编辑 `config/rules.md` | 已有项目，需要调整规则 |
| 项目初始化 | 使用 `/m-init`，AI 会询问规则并写入 | 新项目初始化 |
| 讨论调整 | 使用 `/m-evolve <问题>` 讨论 | 不确定如何调整时 |

**示例**：互联网项目不需要多币种

```markdown
## 项目定位

- **类型**：互联网
- **复杂度**：中

## 设计准则

### 一、数据一致性

- 聚合内强一致，聚合间最终一致

（删除多币种、多汇率章节）
```

## 四、当前项目信息

### 核心领域

| 领域 | 类型 | 说明 |
|------|------|------|
| 订单管理 | 核心域 | 客户订单全生命周期管理 |
| 生产管理 | 核心域 | 生产计划、进度跟踪、质量控制 |
| 采购管理 | 支撑域 | 面辅料采购、供应商管理 |
| 库存管理 | 支撑域 | 仓储、出入库管理 |
| 财务管理 | 支撑域 | 收付款、成本核算、多币种结算 |
| 客户管理 | 通用域 | 客户信息、信用管理 |
| 基础数据 | 通用域 | 汇率、币种、组织架构等 |

### 快速开始

```bash
# 设计订单管理领域
/m-design 订单管理

# 设计客户管理领域
/m-design 客户管理

# 批量应用方案
/m-apply
```

## 五、项目结构

```
config/
  rules.md                # 设计准则（DDD 建模规约）
  roles.md                # 角色定义（参与设计的角色）

solution/
  {domain}.md             # 待确认方案（角色评审后）

design/
  domain/
    {domain}/
      model.md            # 领域模型（实体、聚合、状态机、规则）
  model.md                # 领域模型汇总（统一术语表、领域关系图）

reflect/
  YYYY-MM-DD-{主题}.md    # 反思记录
```

## 六、设计原则

| 规则 | 说明 |
|------|------|
| 角色必须参与 | 每个设计方案必须经过角色评审 |
| 用户原话保留 | 用户原始描述必须保留在方案中 |
| 待定任务记录 | 不确定项必须记录到待定清单 |
| 用户确认 | 更新设计前必须经用户确认 |
| 聚合间引用 | 通过 ID 引用，不直接持有对象 |
