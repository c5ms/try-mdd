> 日期：2026-04-22
> 领域：采购合同

## 一、错误梳理

| # | 错误描述 | 发现时机 | 用户指出方式 | 改正方式 |
|---|---------|---------|------------|---------|
| 1 | 状态压扁成单一枚举 `status`，违反正交分解规则 | 初版设计完成后 | 用户直接质疑："只有一个状态属性码？这些状态都是非此即彼的码？不存在重叠的情况码？" | 拆分为四个正交维度：`approvalStatus`、`contractStatus`、`receivingStatus`、`settlementStatus` |

---

## 二、根因分析

| 根因类型 | 涉及错误 | 具体说明 |
|---------|---------|---------|
| **自检执行不严格** | #1 | 建模元规则第一条"状态正交分解"明确要求检查是否存在"并且"连接的双状态，但自检时直接标记"✓ 状态为单一维度流转，无正交状态"，未认真执行测试 |
| **推理跳跃** | #1 | 未测试状态组合（如"正在执行并且部分入库"是否有意义），直接假设状态链是合理的 |

## 五、本次错误详情

### 错误设计（初版）

```
status: DRAFT | PENDING_APPROVAL | PENDING_SUPPLIER_CONFIRM | EXECUTING | RECEIVED | SETTLED | CANCELLED
```

**问题**：入库进度、结算进度、审批状态、合同状态全部"压扁"成一个状态枚举。

### 正确设计（修正后）

| 维度 | 字段名 | 状态值 |
|------|--------|--------|
| 审批状态 | `approvalStatus` | DRAFT / PENDING_APPROVAL / APPROVED / CANCELLED |
| 合同状态 | `contractStatus` | NONE / PENDING / CONFIRMED / REJECTED |
| 入库进度 | `receivingStatus` | NONE / PARTIAL / FULL |
| 结算进度 | `settlementStatus` | NONE / PARTIAL / FULL |

### 测试过程

| 测试句式 | 是否有意义 | 结论 |
|----------|------------|------|
| "正在执行 并且 部分入库" | ✓ 有意义 | 入库进度应为独立维度 |
| "已入库 并且 未结算" | ✓ 有意义 | 结算进度应为独立维度 |
| "草稿 并且 已入库" | ✗ 无意义（胡话） | 跨维度不变式封死 |
| "待审批 并且 合同已确认" | ✗ 无意义（胡话） | 跨维度不变式封死 |

## 六、经验教训

| 教训 | 说明 |
|------|------|
| 自检不能只写结论 | 必须展示测试过程，否则容易"假执行" |
| 状态设计必须测试组合 | 用"并且"连接测试，发现正交维度 |
| 用户质疑是关键信号 | 用户问"是否存在重叠"时，说明设计有问题 |
